How should skeleton loading work?

Three approaches — different loading experiences. Pick the one that feels right.

A
Full-page skeleton Recommended
app/loading.tsx — entire page replaced by a skeleton while data loads, then swaps in
⏳ While loading…
✅ After load — full page appears
✓ Simplest to build — one file, no refactoring
✓ Perfectly matches actual page layout
✗ Whole page is skeleton until everything loads
✗ Header/nav also disappears (unless in layout)
B
Section streaming
Suspense boundaries per section — header loads instantly, sections appear one by one
⏳ While loading…
Header instantly visible
✅ Sections pop in as ready
Header
✓ Header visible immediately
✓ Sections progressively load
✗ Requires refactoring page into async sub-components
✗ getStats() called once but needs React.cache() wiring
C
loading.tsx + section skeletons
Route-level loading.tsx skeleton that mirrors every section precisely
⏳ Precise mirror of real layout
✓ Pixel-accurate skeleton of real UI
✓ No refactoring needed
✓ Fast to build — pure HTML/CSS
✗ Skeleton and real layout can drift if UI changes
✗ Still replaces whole page (same as A but more detailed)
vs Option A: Same mechanism (loading.tsx), but Option C builds a more detailed, faithful skeleton that closely mirrors every card, chart, and strip. Option A is a simpler, faster placeholder.
